이 문서는 `v0.1.0-mvp`로 고정된 첫 플레이어블을 기준으로, 후속 버전의 핵심 모드 확장을 정의한다.

MVP의 기존 5-Day Run은 **자유모드**로 명명한다.  
새 후속 핵심 모드는 **의뢰 모드**다.

의뢰 모드는 장중 단발 미션이 아니라, 일정 기간 안에 의뢰주가 요구한 가격 조건을 달성하거나 유지하는 기간형 계약 모드다. 플레이어는 고정 보상을 받는 대신, 가능한 적은 예산과 낮은 리스크로 조건을 달성해야 한다.

| 버전 | 변경 내용 |

## 1. 제품 방향

### 1.1 MVP Baseline

`v0.1.0-mvp`는 다음 역할을 가진다.

1. 현재 구현된 5-Day Run을 안정 baseline으로 고정한다.
2. 이 모드를 후속 버전에서 **자유모드**라고 부른다.
3. 새 기능은 baseline 위에 별도 모드나 확장 규칙으로 추가한다.
4. 자유모드의 핵심 루프, 수동 액션, MADNESS, 마켓 대시보드, VALUE, Day/Final Settlement는 그대로 유지한다.

### 1.2 후속 제품 목표

v0.2.0의 제품 목표는 두 가지 플레이 감각을 분리하는 것이다.

| 모드 | 핵심 판타지 | 핵심 점수 |
| --- | --- | --- |
| 자유모드 | 플레이어가 직접 종목을 고르고 본인의 판을 만든다 | 생존, 수익, 감시 회피, 장기 운용 |
| 의뢰모드 | 의뢰주가 원하는 가격 조건을 계약 기간 안에 맞춘다 | 고정 보상 대비 최소 비용, 조건 달성률, 리스크 관리 |

자유모드는 “내가 먹기 위해 판을 짜는 모드”다.  
의뢰모드는 “남의 계약을 가장 효율적으로 처리하는 모드”다.

## 2. 자유모드

자유모드는 현재 MVP의 정식 명칭이다.

### 2.1 핵심 규칙

1. 플레이어가 Run 시작 시 섹터와 종목을 직접 선택한다.
2. 5일 동안 매일 개장 전 카드, 뉴스, 장중 운용, 정산을 진행한다.
3. 목표는 높은 수익, 낮은 감시도, 예산 보전, 생존이다.
4. 플레이어의 포지션과 평균단가, 예산, MADNESS, VALUE 순위가 핵심 판단 기준이다.
5. 자유모드는 목표 가격이 외부에서 강제되지 않는다.

### 2.2 현행 시스템 유지

자유모드는 다음 MVP 시스템을 유지한다.

1. 5-Day Run
2. Day당 3분 장중 운용
3. 허구 섹터 8개, 허구 종목 24개
4. 개장 전 카드 4개
5. 수동 액션 4개
6. 자동 카드 8개
7. 문서 이벤트 8개
8. Retail Swarm
9. MADNESS
10. 마켓 대시보드 VALUE와 순위 피드백
11. Day/Final Settlement

---

## 3. 의뢰모드

의뢰모드는 기간형 계약 미션이다.

플레이어는 의뢰주가 제시한 종목, 기간, 가격 조건, 리스크 조건, 고정 보상을 보고 계약을 수락한다. 계약 후에는 자유모드와 같은 도구를 사용하지만, 목표는 개인 수익 극대화가 아니라 **계약 조건을 가장 적은 비용으로 달성하는 것**이다.

### 3.1 핵심 차별점

| 항목 | 자유모드 | 의뢰모드 |
| --- | --- | --- |
| 시작 동기 | 플레이어의 자체 운용 | 외부 의뢰 계약 |
| 대상 종목 | 플레이어가 직접 선택 | 의뢰가 지정하거나 제한 |
| 목표 | 수익과 생존 | 기간형 가격 조건 달성 |
| 보상 | 포지션 수익 중심 | 고정 계약 보상 중심 |
| 점수 | 누적 수익, 감시등급 | 보상 - 투입 비용 - 리스크 비용 |
| 정보 카드 | 종목분석 | 전문가 리포트 |

### 3.2 의뢰 기간

의뢰는 장중 하나의 이벤트가 아니라 Day 단위 기간을 가진다.

가능한 기간 예시는 다음과 같다.

1. `D+1 안에 조건 달성`
2. `D+3 안에 목표가 터치`
3. `5일 동안 밴드 유지`
4. `마지막 Day 종가 조건 달성`
5. `목표 터치 후 남은 기간 방어`

MVP baseline이 5-Day Run이므로, 의뢰모드의 1차 범위도 1~5 Day 계약을 기본으로 한다.

## 4. 의뢰주 유형

의뢰주는 목표 방향과 리스크 성향을 결정한다.

| 의뢰주 유형 | 의도 | 대표 목표 |
| --- | --- | --- |
| `long_holder` | 보유 물량을 유리한 가격대에서 정리하려는 의뢰주 | 상승 터치, 상단 밴드 유지, 종가 상단 마감 |
| `short_seller` | 하락에서 이익을 보려는 의뢰주 | 하락 터치, 상단 돌파 방지, 낮은 종가 |
| `accumulator` | 싸게 모으기 위해 가격을 특정 밴드 아래에 두려는 의뢰주 | 상단 제한, 저가 밴드 유지, 과열 방지 |
| `defender` | 담보, 평판, 계약 조건 때문에 하단 이탈을 막으려는 의뢰주 | 하단 방어, 특정 종가 이상 유지 |
| `pump_exit` | 단기간 관심과 거래대금을 만든 뒤 빠져나가려는 의뢰주 | 급등 터치, VALUE 순위 상승, 이후 밴드 방어 |

의뢰주 유형은 UI에서 실제 금융 행위처럼 구체적으로 설명하지 않는다.  
플레이어에게는 “의뢰 성향”, “목표 방향”, “리스크 성향” 같은 게임 용어로 표시한다.

## 5. 의뢰 목표 조건

의뢰 목표는 하나 이상의 조건으로 구성된다.

### 5.1 목표 타입

| 타입 | 설명 | 예시 |
| --- | --- | --- |
| `touch` | 기간 안에 특정 가격 또는 밴드를 한 번 터치 | `D+3 안에 14,000원 터치` |
| `maintain` | 특정 기간 동안 가격을 밴드 안에 유지 | `3일 동안 12,500~13,200원 유지` |
| `close_above` | 특정 Day 종가가 기준 이상 | `D+2 종가 11,800원 이상` |
| `close_below` | 특정 Day 종가가 기준 이하 | `D+4 종가 8,800원 이하` |
| `close_inside_band` | 특정 Day 종가가 밴드 안 | `마지막 Day 종가 10,900~11,300원` |
| `never_break` | 기간 동안 하단 또는 상단 이탈 금지 | `5일 동안 9,500원 아래 이탈 금지` |
| `rank` | 마켓 대시보드 VALUE 순위 조건 | `마지막 Day VALUE TOP 5` |
| `value` | 누적 VALUE 조건 | `D+3까지 누적 VALUE 300M 이상` |
| `touch_then_maintain` | 목표 터치 후 남은 기간 밴드 방어 | `D+2 안에 14,000원 터치 후 13,500~14,300원 유지` |

### 5.2 방향성

의뢰는 상승만 요구하지 않는다.

| 방향 | 설명 |
| --- | --- |
| `upward` | 목표가 상향, 상단 터치, 높은 종가 |
| `downward` | 목표가 하향, 하단 터치, 낮은 종가 |
| `range` | 밴드 안에 가격 유지 |
| `defense` | 특정 하단 또는 상단 방어 |
| `attention` | VALUE, 관심, 순위 조건 달성 |
| `stealth` | 낮은 감시도와 낮은 과열을 유지하며 조건 달성 |

하락 의뢰에서는 매도 압력만 강하게 넣으면 오히려 VALUE, MADNESS, 반발 매수세가 올라 목표 달성이 어려워질 수 있다.  
따라서 의뢰모드는 도구를 무조건 강하게 쓰는 게임이 아니라, 시장 반응을 보며 압력, 유동성, 관심도, 감시도를 조합하는 퍼즐이어야 한다.

## 6. 도구와 전략성

의뢰모드는 자유모드의 기존 도구를 재사용한다.

### 6.1 상승 의뢰에 유리한 도구

1. 매수봇
2. 유동성 공급
3. 전문가 리포트 상향
4. 마켓 대시보드 VALUE 순위 상승
5. MADNESS 유도 후 Retail Swarm 유입

### 6.2 하락 의뢰에 유리한 도구

1. 매도봇
2. 포지션 정리
3. 전문가 리포트 하향
4. 과열 억제
5. VALUE 순위 과상승 방지

### 6.3 하락 의뢰의 핵심 리스크

하락 의뢰는 단순히 가격을 누르는 것보다 더 어렵게 설계한다.

1. 너무 빠른 하락은 반발 매수세를 만든다.
2. 큰 거래대금은 대시보드 순위를 올려 외부 관심을 부른다.
3. MADNESS가 높으면 포지션 정리 매도 물량도 Retail Swarm이 흡수할 수 있다.
4. 유동성이 너무 낮으면 가격은 쉽게 움직이지만 변동성과 감시 리스크가 커진다.
5. 전문가 리포트가 목표 방향의 명분을 만들면 같은 가격 이동에도 감시 리스크가 낮아질 수 있다.

## 7. 전문가 리포트

의뢰모드에서는 자유모드의 `종목분석` 역할을 `전문가 리포트`로 바꾼다.

전문가 리포트는 의뢰 목표를 직접 노출하지 않고, 목표 근처의 그럴듯한 가격 전망이나 박스권을 제시한다.

### 7.1 리포트 역할

1. 목표 price band에 가까운 기준 가격 또는 범위를 제시한다.
2. Retail Swarm의 기대 가격, MADNESS, VALUE 관심도에 영향을 준다.
3. 목표가 터무니없을수록 리포트 신뢰도는 낮아진다.
4. 신뢰도가 낮은 리포트는 감시도와 사회적 비용 리스크를 높인다.
5. 상승, 하락, 박스권, 방어형 리포트가 모두 가능하다.

### 7.2 예시

| 의뢰 조건 | 전문가 리포트 표시 방향 |
| --- | --- |
| `D+4 안에 14,500원 터치` | `단기 적정가 13,800~15,200원` |
| `5일간 11,000~11,600원 유지` | `박스권 10,900~11,800원 전망` |
| `D+3 안에 8,500원 이하 터치` | `단기 하방 재평가 구간 8,200~9,000원` |
| `9,500원 아래 이탈 금지` | `하단 방어 가능 구간 9,400~10,200원` |

리포트 문구는 실제 투자 리포트처럼 보이지 않도록 허구 섹터, 허구 종목, 추상 지표 중심으로 작성한다.

## 8. 보상과 난이도

의뢰 보상은 계약 시작 시 고정된다.

성공 시 플레이어는 고정 보상을 받는다.  
하지만 최종 평가는 보상 총액이 아니라 **얼마나 적은 비용과 낮은 리스크로 달성했는가**를 본다.

### 8.1 보상 증가 요인

보상은 다음 요소가 어려울수록 비선형으로 증가한다.

1. 현재가에서 목표가까지의 거리
2. 목표 밴드의 좁음
3. 계약 기간의 짧음
4. 유지 기간의 김
5. 종목 체급과 영향력 저항
6. 상승/하락 방향과 현재 시장 분위기의 불일치
7. VALUE, 감시도, MADNESS 제한 조건
8. 복수 조건 조합

터무니없는 목표가는 보상이 기하급수적으로 커진다.  
대신 리포트 신뢰도 저하, 감시도 상승, Retail Swarm 반작용, 실패 확률도 함께 커진다.

### 8.2 정산 기준

의뢰모드 최종 점수는 다음 기준을 따른다.

```text
 계약 고정 보상
- 투입 예산
- 감시 리스크 비용
- 사회적 비용
- 미달성 패널티
- 과도한 VALUE/MADNESS 부작용
= 계약 순성과
```

성공 여부와 별개로, 같은 의뢰를 더 적은 비용으로 달성한 플레이가 더 높은 평가를 받는다.

## 9. 상태 모델 초안

정확한 스키마는 SRS에서 확정한다. PRD 수준에서는 다음 개념을 요구한다.

```ts
type GameMode = "free" | "contract";

type ContractSponsorType =
  | "long_holder"
  | "short_seller"
  | "accumulator"
  | "defender"
  | "pump_exit";

type ContractDirection =
  | "upward"
  | "downward"
  | "range"
  | "defense"
  | "attention"
  | "stealth";

type ContractObjective =
  | { type: "touch"; targetPrice: number; deadlineDay: number }
  | { type: "maintain"; lowerPrice: number; upperPrice: number; requiredDays: number }
  | { type: "close_above"; targetPrice: number; day: number }
  | { type: "close_below"; targetPrice: number; day: number }
  | { type: "close_inside_band"; lowerPrice: number; upperPrice: number; day: number }
  | { type: "never_break"; lowerPrice?: number; upperPrice?: number; durationDays: number }
  | { type: "rank"; maxRank: number; deadlineDay: number }
  | { type: "value"; minValue: number; deadlineDay: number }
  | {
      type: "touch_then_maintain";
      targetPrice: number;
      touchDeadlineDay: number;
      lowerPrice: number;
      upperPrice: number;
      maintainDays: number;
    };

type ContractMandate = {
  id: string;
  sponsorType: ContractSponsorType;
  direction: ContractDirection;
  assetId: string;
  durationDays: number;
  objectives: ContractObjective[];
  fixedReward: number;
  riskLevel: number;
  reportConfidence: number;
};
```

## 10. UI 요구사항

### 10.1 메인 메뉴

메인 메뉴에는 모드 선택을 추가한다.

1. 자유모드
2. 의뢰모드

자유모드는 기존 MVP 시작 흐름으로 진입한다.  
의뢰모드는 계약 선택 화면으로 진입한다.

### 10.2 의뢰 선택 화면

의뢰 선택 화면은 다음 정보를 표시한다.

1. 의뢰 성향
2. 대상 섹터와 종목
3. 계약 기간
4. 목표 조건 요약
5. 고정 보상
6. 예상 리스크
7. 전문가 리포트 신뢰도

목표 조건은 플레이어가 이해할 수 있어야 하지만, 실제 금융 절차처럼 설명하지 않는다.

### 10.3 의뢰 브리핑

의뢰 브리핑은 다음 정보를 제공한다.

1. 목표 bandline 또는 threshold
2. 남은 Day
3. 성공/실패 조건
4. 계약 보상
5. 현재 예상 난이도
6. 전문가 리포트 요약

### 10.4 장중 화면

의뢰모드 장중 화면은 자유모드 UI를 재사용하되 다음 요소를 추가한다.

1. 계약 목표 tracker
2. 목표 bandline 차트 overlay
3. 터치 달성 여부
4. 유지 조건 진행률
5. 계약 순성과 추정치
6. 전문가 리포트 신뢰도

### 10.5 정산 화면

Day Settlement는 해당 Day의 계약 진행 상황을 보여준다.

Final Contract Settlement는 다음을 보여준다.

1. 계약 성공/실패
2. 각 objective 달성 여부
3. 고정 보상
4. 총 투입 예산
5. 감시/사회적 비용
6. 계약 순성과
7. 효율 등급

## 13. 수용 기준

v0.2.0 의뢰모드의 최소 수용 기준은 다음이다.

1. 플레이어가 메인 메뉴에서 자유모드와 의뢰모드를 구분해 시작할 수 있다.
2. 자유모드는 `v0.1.0-mvp` baseline과 같은 핵심 루프를 유지한다.
3. 의뢰모드는 최소 4개 이상의 계약 샘플을 제공한다.
4. 계약 샘플에는 상승, 하락, 밴드 유지, 하단/상단 방어 유형이 포함된다.
5. 계약은 1~5 Day 기간을 가진다.
6. 계약 objective는 Day 진행과 장중 가격/VALUE 상태를 기준으로 평가된다.
7. 전문가 리포트는 의뢰모드에서 목표 방향을 간접적으로 보조한다.
8. 성공 시 고정 보상이 지급된다.
9. 최종 평가는 보상뿐 아니라 투입 예산, 감시도, 사회적 비용을 함께 계산한다.
10. 하락 의뢰는 과도한 VALUE, MADNESS, 반발 매수세 리스크를 가져야 한다.
11. 의뢰모드도 실제 금융 데이터나 실제 금융 절차를 사용하지 않는다.

---

## 14. 안전 표현 원칙

의뢰모드는 소재상 “가격을 움직이는 계약”을 다루지만, 실제 금융 범죄 절차를 설명하는 제품이 아니다.

따라서 다음 원칙을 유지한다.

1. 모든 종목, 섹터, 뉴스, 의뢰주는 허구다.
2. 실제 시장 데이터, 실제 거래소, 실제 종목명을 쓰지 않는다.
3. 실제 주문 방식, 회피 방식, 타이밍 기법을 설명하지 않는다.
4. UI 문구는 `시장 압력`, `가격 밴드`, `전문가 리포트`, `관심도`, `VALUE`, `리스크` 같은 추상 게임 용어를 사용한다.
5. 하락 의뢰도 실제 공매도 전략 안내가 아니라, 허구 가격 조건을 맞추는 게임 규칙으로 표현한다.
6. 정산에는 감시도와 사회적 비용을 명확히 포함한다.

---

## 15. 개발 메모

의뢰모드는 자유모드 위에 얹는 것이 아니라, 자유모드와 같은 엔진을 공유하는 별도 목표 구조로 설계한다.

구현 우선순위는 다음과 같다.

1. objective evaluator를 먼저 만든다.
2. contract settlement를 만든다.
3. UI 문구와 tracker를 얹는다.
4. 전문가 리포트는 처음에는 수치 효과가 작은 보조 카드로 시작한다.
5. 하락 의뢰의 반발 매수세는 기존 MADNESS, VALUE 순위, Retail Swarm 로직을 재사용한다.

핵심은 새 액션을 많이 추가하는 것이 아니라, 기존 도구의 의미가 계약 조건에 따라 달라지게 만드는 것이다.
