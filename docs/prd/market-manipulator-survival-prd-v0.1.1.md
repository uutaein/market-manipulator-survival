### 0.1 현재 확정 기준

2026-06-13 기준 현재 제품 방향은 다음과 같다.

1. 게임은 JS 또는 TypeScript 기반의 경량 2D 브라우저 게임으로 만든다.
2. 1차 구현은 데스크톱 브라우저 실행을 우선한다.
3. 후속 버전에서 Electron 패키징을 검토할 수 있다.
4. 플레이어는 가상의 주식/코인 시장에서 활동하는 “세력 운용자” 역할을 맡는다.
5. 실제 종목, 실제 거래소, 실제 가격 데이터, 실제 투자 API는 사용하지 않는다.
6. 게임은 금융 범죄 방법을 설명하지 않고, 시장 조작을 추상화된 풍자 시스템으로 표현한다.
7. 핵심 플레이는 Vampire Survivors처럼 자동 발동되는 능력 조합과 생존 루프를 따른다.
8. 화면과 UX는 Papers, Please처럼 문서, 도장, 감시 보고서, 제한 시간 선택, 관료제적 압박감을 중심으로 구성한다.
9. 일반적인 필드 맵 대신 차트, 호가창, 뉴스 티커, 감시국 문서, 작전 카드 패널이 전장 역할을 한다.
10. 한 판은 10~20분 내 종료되는 짧은 세션을 목표로 한다.
11. MVP는 싱글플레이어 오프라인 게임으로 만든다.
12. 결과 기록은 P0에서 localStorage에 저장한다.

### 0.2 v0.1.1 모호성 해소 결과

PRD v0.1에서 구현 착수 전 모호했던 항목은 다음 기준으로 정리한다.

1. 기술 스택은 **TypeScript + Phaser 3 + Vite**로 확정한다.
2. 순수 JavaScript는 학습용 프로토타입에는 허용하지만, P0 구현 기준은 TypeScript다.
3. 렌더링은 **Phaser 단일 렌더링**을 우선한다. 차트, 카드, 압박 유닛, 문서 이벤트도 Phaser Scene 안에서 처리한다.
4. HTML DOM 오버레이는 개발 편의용 디버그 패널과 접근성 보조 영역에만 제한한다.
5. P0 세션 길이는 **10분 단일 모드**로 확정한다.
6. P0 종목은 허구 종목 3개로 제한한다.
7. P0 작전 카드는 8개로 확정한다.
8. P0 압박 유닛은 5종으로 확정한다.
9. P0 문서 이벤트는 8개로 확정하되, 각 이벤트는 2~3개 선택지만 가진다.
10. 저장 방식은 P0에서 **localStorage**로 확정한다.
11. localStorage에는 설정, 최고 기록, 최근 정산 기록만 저장한다.
12. 세션 중 상세 로그와 리플레이 저장은 P0 범위에서 제외한다.
13. 승리 조건은 **10분 생존 + 목표 가격 밴드 도달 또는 유지**로 확정한다.
14. 실패 조건은 **감시 위험도 100 도달, 유동성 0 도달, 가격 붕괴선 이탈**로 확정한다.
15. 점수는 수익률, 감시 위험도 잔여분, 유동성 잔여분, 사회적 비용 패널티를 합산한다.
16. UI 톤은 Papers, Please의 원작 요소를 복제하지 않고, **거래소 감시국/작전 보고서/차트 작전실** 정체성으로 재해석한다.
17. 법적·윤리적 고지는 메인 메뉴, 로딩 화면, 결과 정산 화면에 짧게 노출한다.
18. 모든 작전명과 효과 설명은 실제 금융 범죄 절차를 설명하지 않는 추상 표현으로 작성한다.
19. “세력 성공”을 영웅화하지 않고, 결과 정산에서 후폭풍과 사회적 비용을 함께 표시한다.
20. 위 결정의 근거와 변경 영향은 `ADR-0001`에 기록한다.

## 1. 배경

주식과 코인 시장은 실제로 많은 사람에게 숫자, 차트, 뉴스, 소문, 군중 심리, 탐욕, 공포가 섞인 복잡한 공간으로 인식된다.

게임 관점에서 이 소재는 다음 이유로 흥미롭다.

1. 차트는 시간에 따라 계속 움직이는 자연스러운 전장이다.
2. 가격, 유동성, 여론, 감시 위험도는 서로 충돌하는 자원으로 만들기 쉽다.
3. “세력”, “개미”, “고래”, “감시”, “찌라시”, “패닉셀” 같은 단어는 이미 강한 게임적 상상력을 가진다.
4. 그러나 실제 조작 방법을 자세히 재현하면 윤리적·법적 리스크가 크다.
5. 따라서 이 게임은 실제 금융 행위를 시뮬레이션하기보다, 시장의 탐욕 구조를 과장된 블랙코미디로 표현해야 한다.

기존 Vampire Survivors류 게임은 자동 공격, 웨이브 압박, 레벨업 선택, 빌드 조합이 강점이다.  
이 구조를 차트와 시장 압박으로 바꾸면 “몬스터를 피하는 게임”이 아니라 “숫자와 위험을 버티는 게임”이 된다.

Papers, Please류 감성은 단순 액션보다 문서, 규정, 심사, 책임, 선택의 후폭풍을 강조한다.  
이를 시장 감시 보고서와 거래소 질의서 형태로 바꾸면, 플레이어는 단순히 수익만 보는 것이 아니라 “들키지 않을 것인가”, “어떤 문서에 어떤 도장을 찍을 것인가”를 고민하게 된다.

## 2. 목표

이 제품의 목표는 **차트를 전장으로 삼고, 작전 카드를 무기로 쓰며, 감시 문서를 압박 요소로 사용하는 경량 2D 시장 풍자 생존 게임**을 만드는 것이다.

### 2.1 핵심 목표

1. 데스크톱 브라우저에서 가볍게 실행되는 2D 게임을 만든다.
2. 한 판이 짧고 반복 가능한 오토 서바이벌 루프를 제공한다.
3. 플레이어가 가격, 유동성, 여론, 감시 위험도를 동시에 관리하게 한다.
4. 작전 카드는 자동 발동되며, 플레이어는 레벨업 때 카드 선택과 강화 방향을 결정한다.
5. 시장 압박 요소는 몬스터 대신 패닉셀러, 단타봇, 경쟁 세력, 감시봇, 언론 이벤트로 표현한다.
6. Papers, Please풍의 문서 이벤트를 통해 제한 시간 선택과 후폭풍을 만든다.
7. 게임 내 모든 종목과 사건은 허구로 구성한다.
8. 실제 금융 범죄 절차를 안내하지 않는 추상적 수치 시스템으로 설계한다.
9. MVP는 10분 생존 라운드 1개를 완성도 있게 플레이할 수 있게 한다.
10. 추후 종목, 카드, 문서 이벤트, 엔딩을 데이터 기반으로 확장할 수 있게 한다.

### 2.2 제품이 해결하려는 핵심 재미 문제

> 이 게임의 핵심 재미는 “실제 돈을 버는 것”이 아니라, **탐욕스러운 숫자 게임을 하다가 감시와 혼란이 폭주하는 블랙코미디적 압박감**이다.

따라서 1차 제품은 복잡한 현실 경제 재현보다 다음에 집중한다.

1. 가격을 밀어 올리거나 방어하는 감각이 있는가?
2. 자동 발동 카드 조합이 매 판 다르게 느껴지는가?
3. 유동성과 감시 위험도 사이의 선택이 긴장감을 만드는가?
4. 문서 이벤트가 단순 팝업이 아니라 플레이 판단을 흔드는가?
5. 결과 정산이 플레이어의 선택을 풍자적으로 되돌려주는가?

## 3. Non-Goals

이번 1차 범위에서는 아래 기능을 포함하지 않는다.

1. 실제 주식 또는 코인 API 연동
2. 실제 종목명, 실제 거래소명, 실제 인물명 사용
3. 실제 가격 데이터, 뉴스 데이터, 온체인 데이터 사용
4. 실제 시세조종 절차, 주문 방식, 불법 행위 방법의 구체적 재현
5. 실제 투자, 거래, 매매, 지갑 연결 기능
6. 현금 결제, 토큰, NFT, 리워드형 수익 구조
7. 멀티플레이
8. 온라인 랭킹 서버
9. 모바일 앱 개발
10. 콘솔 게임기 포팅
11. 고해상도 일러스트 중심의 대형 아트 제작
12. 복잡한 거시경제 시뮬레이션
13. 실제 법률 자문 또는 금융 교육 콘텐츠
14. AI 기반 실시간 스토리 생성
15. 모드 마켓 또는 유저 제작 콘텐츠 공유 서버
16. 클라우드 세이브
17. 계정 시스템
18. 광고 수익화 시스템
19. 실제 Papers, Please의 UI, 문구, 세계관, 캐릭터, 국가 설정 복제
20. 실제 Vampire Survivors의 캐릭터, 무기, 적, 레벨 구조 복제

> 1차 목표는 대형 경제 시뮬레이터가 아니라, 차트와 문서 UI를 이용한 작고 강한 생존형 풍자 게임이다.

## 4. 플레이 대상과 용어

### 4.1 Market Session

Market Session은 한 판의 플레이 단위다.

각 세션은 하나의 가상 종목, 제한 시간, 시장 성향, 승리 조건, 위험 조건을 가진다.

| 항목 | 설명 |
| --- | --- |
| 세션 ID | 한 판을 구분하는 내부 식별자 |
| 종목 이름 | 허구의 주식 또는 코인 이름 |
| 시장 유형 | 주식형, 코인형, 밈테마형, 작전주형 등 |
| 제한 시간 | MVP 기본 10분 |
| 시작 가격 | 차트의 초기 가격 |
| 목표 가격 | 성공 조건에 필요한 가격 |
| 초기 유동성 | 작전 실행 자원 |
| 초기 여론 | 시장 반응 수치 |
| 초기 감시 위험도 | 규제 및 거래소 감시 수치 |
| 난이도 | 초보, 보통, 위험 등 |

### 4.2 Market Board

Market Board는 게임의 메인 전장이다.

일반적인 맵 대신 차트, 주문 흐름, 뉴스 티커, 감시 알림, 문서 팝업이 전장 역할을 한다.

| 영역 | 설명 |
| --- | --- |
| 중앙 | 실시간 가격 차트 |
| 좌측 | 보유 작전 카드와 자동 발동 상태 |
| 우측 | 가격, 유동성, 여론, 감시 위험도 |
| 하단 | 뉴스 티커, 경고, 이벤트 로그 |
| 팝업 | 감시국 문서, 거래소 질의서, 내부 보고서 |

### 4.3 Operation Card

Operation Card는 자동으로 발동되는 능력이다.

플레이어가 직접 연타하지 않아도 카드마다 정해진 주기와 조건에 따라 효과가 발생한다.

| 항목 | 설명 |
| --- | --- |
| 카드 이름 | 작전 카드 표시명 |
| 레벨 | 강화 단계 |
| 발동 주기 | 자동 효과가 발생하는 간격 |
| 주요 효과 | 가격, 여론, 유동성, 감시 위험도 변화 |
| 부작용 | 감시 증가, 유동성 소모, 변동성 증가 등 |
| 태그 | 여론형, 방어형, 변동성형, 은폐형 등 |

### 4.4 Pressure Unit

Pressure Unit은 플레이어를 압박하는 위협 요소다.

몬스터처럼 충돌하는 물리 적이 아니라, 차트와 수치에 악영향을 주는 시장 이벤트로 표현한다.

| 위협 | 설명 |
| --- | --- |
| 패닉셀러 | 가격 하방 압력을 만든다 |
| 단타봇 | 변동성을 키우고 예측을 어렵게 만든다 |
| 고래 추적자 | 플레이어 작전 효율을 낮춘다 |
| 경쟁 세력 | 반대 방향의 가격 압력을 만든다 |
| 거래소 감시봇 | 감시 위험도를 올린다 |
| 언론 특보 | 여론을 급변시킨다 |

### 4.5 Inspection Event

Inspection Event는 문서 기반 선택 이벤트다.

플레이어는 제한 시간 안에 문서를 읽고 선택지 중 하나를 골라야 한다.

| 항목 | 설명 |
| --- | --- |
| 문서 제목 | 거래소 질의서, 감시 보고서, 내부 메모 등 |
| 발동 조건 | 감시 위험도, 시간, 카드 사용, 가격 급등락 등 |
| 제한 시간 | 선택 가능한 시간 |
| 선택지 | 소명, 무시, 작전 축소, 역공, 은폐 등 |
| 즉시 효과 | 자원 변화 |
| 지연 효과 | 후속 이벤트 또는 위험 증가 |

### 4.6 Settlement Report

Settlement Report는 라운드 종료 후 결과 화면이다.

수익뿐 아니라 피해, 조사 위험, 시장 혼란, 플레이어 평판을 풍자적으로 보여준다.

## 6. 핵심 사용자 시나리오

### 6.1 첫 세션 시작

사용자로서, 게임을 실행한 뒤 바로 가상의 종목을 선택하고 장을 시작하고 싶다.  
그래야 복잡한 설정 없이 핵심 루프를 경험할 수 있다.

### 6.2 작전 카드 선택

사용자로서, 시작 카드 또는 레벨업 카드 중 하나를 선택하고 싶다.  
그래야 매 판 다른 빌드를 만들 수 있다.

### 6.3 차트 압박 대응

사용자로서, 가격이 흔들리고 시장 압박이 커지는 상황에서 작전 방향을 조정하고 싶다.  
그래야 단순 방치가 아니라 선택의 긴장감을 느낄 수 있다.

### 6.4 유동성 관리

사용자로서, 강한 카드를 많이 쓰면 유동성이 줄어드는 것을 보고 싶다.  
그래야 무조건 강한 선택만 하는 것이 아니라 자원 관리를 고민하게 된다.

### 6.5 감시 위험도 관리

사용자로서, 공격적인 작전이 감시 위험도를 올리는 것을 보고 싶다.  
그래야 수익과 안전 사이에서 선택하게 된다.

### 6.6 문서 이벤트 처리

사용자로서, 거래소 질의서나 감시 보고서가 뜨면 제한 시간 안에 대응을 고르고 싶다.  
그래야 Papers, Please식 압박감을 느낄 수 있다.

### 6.7 라운드 성공

사용자로서, 제한 시간까지 목표 가격을 유지하거나 수익 목표를 달성하고 싶다.  
그래야 내 빌드와 선택이 성공했다는 성취감을 얻을 수 있다.

### 6.8 라운드 실패

사용자로서, 감시 위험도 폭주, 유동성 고갈, 가격 붕괴로 실패할 수 있어야 한다.  
그래야 위험 관리가 실제 플레이 의미를 가진다.

### 6.9 결과 보고서 확인

사용자로서, 라운드 종료 후 수익뿐 아니라 시장 혼란, 조사 위험, 평판을 보고 싶다.  
그래야 게임의 풍자 톤이 정산 화면까지 이어진다.

## 7. 핵심 기능

### 7.1 메인 메뉴

메인 메뉴는 빠른 시작을 우선한다.

필요한 항목은 다음과 같다.

1. 새 게임
2. 기록실
3. 설정
4. 크레딧
5. 종료 또는 브라우저 닫기 안내

MVP에서는 새 게임과 설정만 우선 구현해도 된다.

### 7.2 종목 선택

사용자는 라운드 시작 전 가상 종목을 선택한다.

MVP 종목 후보는 다음과 같다.

| 종목 | 유형 | 특징 |
| --- | --- | --- |
| DUST BIO | 작전주형 | 유동성 낮음, 가격 변동 큼 |
| MOONCAT | 밈코인형 | 여론 효과 큼, 감시 변동 큼 |
| OLD STEEL | 전통주형 | 가격 안정적, 목표 달성 느림 |

각 종목은 실제 회사를 연상시키지 않는 허구 이름을 사용한다.

### 7.3 차트형 전장

메인 플레이 화면은 다음 정보를 표시한다.

1. 현재 가격
2. 목표 가격
3. 남은 시간
4. 유동성
5. 여론
6. 감시 위험도
7. 보유 작전 카드
8. 발동 중인 효과
9. 뉴스 티커
10. 이벤트 로그

차트는 실제 금융 차트처럼 보일 수 있지만, 데이터는 게임 내부 공식으로 생성한다.

### 7.4 자동 작전 카드

작전 카드는 일정 주기로 자동 발동한다.

MVP 카드 후보는 다음과 같다.

| 카드 | 역할 | 기본 효과 | 기본 부작용 |
| --- | --- | --- | --- |
| 찌라시 살포 | 여론형 | 여론 상승 | 감시 소폭 상승 |
| 매수벽 세우기 | 방어형 | 하락 압력 완화 | 유동성 소모 |
| 매도벽 위협 | 통제형 | 변동성 감소 | 여론 하락 가능 |
| 봇 거래 | 지속형 | 작은 가격 압력 반복 | 감시 누적 |
| 인플루언서 계약 | 폭발형 | 여론 급상승 | 감시 급상승 |
| 물량 흡수 | 안정형 | 가격 급락 방어 | 유동성 큰 소모 |
| 패닉 유도 | 변동성형 | 경쟁 세력 약화 | 여론 불안정 |
| 내부자 루머 | 고위험형 | 가격 상승 압력 | 문서 이벤트 확률 증가 |

카드 설명은 실제 조작 방법이 아니라 게임 수치 효과 중심으로 작성한다.

### 7.5 레벨업 선택

플레이어는 경험치 또는 영향력 게이지가 차면 레벨업 선택지를 받는다.

선택지는 3개 중 1개를 고르는 방식으로 한다.

예시는 다음과 같다.

| 선택지 | 효과 |
| --- | --- |
| 찌라시 살포 Lv.2 | 여론 상승량 증가 |
| 매수벽 세우기 Lv.2 | 방어 지속시간 증가 |
| 감시 회피 프로토콜 | 감시 증가량 감소 |
| 유동성 확보 | 현재 유동성 일부 회복 |
| 뉴스 장악 | 다음 언론 이벤트 효과 완화 |

### 7.6 시장 압박 웨이브

시간이 지날수록 압박 요소가 강해진다.

MVP 압박 요소는 다음과 같다.

| 위협 | 효과 |
| --- | --- |
| 패닉셀러 | 가격 하락 압력 |
| 단타봇 | 변동성 증가 |
| 고래 추적자 | 카드 효율 감소 |
| 거래소 감시봇 | 감시 위험도 증가 |
| 경쟁 세력 | 목표 반대 방향 압력 |

압박 요소는 화면상 아이콘, 차트 마커, 경고 패널로 표현한다.

### 7.7 감시 위험도

감시 위험도는 핵심 실패 조건 중 하나다.

규칙은 다음과 같다.

1. 감시 위험도는 0~100 사이 값이다.
2. 공격적인 카드일수록 감시 위험도를 더 많이 올린다.
3. 감시 위험도가 높을수록 문서 이벤트 발생 확률이 오른다.
4. 감시 위험도 70 이상에서는 경고 UI와 사운드를 강화한다.
5. 감시 위험도 100에 도달하면 라운드는 “조사 착수” 실패로 종료된다.
6. 일부 선택지는 감시 위험도를 낮추지만 가격 또는 수익에 손해를 준다.

### 7.8 유동성

유동성은 작전 실행 자원이다.

규칙은 다음과 같다.

1. 유동성은 0 이상 값이다.
2. 방어형 또는 고성능 카드는 유동성을 소모한다.
3. 유동성이 낮으면 카드 발동 효율이 떨어진다.
4. 유동성이 0이 되면 라운드는 “자금 고갈” 실패로 종료될 수 있다.
5. 특정 레벨업 선택이나 문서 선택으로 유동성을 회복할 수 있다.

### 7.9 여론

여론은 가격 압력과 이벤트 결과에 영향을 준다.

규칙은 다음과 같다.

1. 여론이 높으면 가격 상승 압력이 커진다.
2. 여론이 너무 높으면 감시 위험도 증가량도 커질 수 있다.
3. 여론이 낮으면 패닉셀러 출현 확률이 증가한다.
4. 언론 이벤트는 여론을 크게 흔든다.

### 7.10 문서 이벤트

문서 이벤트는 Papers, Please풍 감성의 핵심 기능이다.

MVP 문서 이벤트 후보는 다음과 같다.

| 이벤트 | 발동 조건 | 선택지 |
| --- | --- | --- |
| 거래소 이상거래 질의서 | 감시 위험도 일정 이상 | 소명 제출 / 무시 / 작전 축소 |
| 익명 제보 메모 | 랜덤 또는 고위험 카드 사용 후 | 묵살 / 내부 조사 / 역이용 |
| 언론 취재 요청 | 여론 급등 시 | 홍보 / 침묵 / 부인 |
| 커뮤니티 폭주 보고서 | 여론 과열 시 | 방치 / 진정 / 선동 |
| 규제기관 예비 경고 | 감시 위험도 80 이상 | 중단 / 축소 / 강행 |

각 선택은 즉시 효과와 지연 효과를 가질 수 있다.

예시:

```text
거래소 이상거래 질의서

최근 비정상적 거래 패턴이 감지되었습니다.
소명 자료를 제출하십시오.

[소명 제출] 감시 -15, 유동성 -10
[무시] 현재 영향 없음, 60초 후 감시 +25
[작전 축소] 가격 압력 -20%, 감시 -25
```

### 7.11 성공 조건

MVP의 기본 성공 조건은 다음 중 하나로 한다.

1. 제한 시간 10분 동안 살아남는다.
2. 제한 시간 종료 시 가격이 목표 가격 이상이다.
3. 감시 위험도와 유동성이 실패 조건에 도달하지 않는다.

정확한 성공 조건은 난이도별로 조정할 수 있다.

### 7.12 실패 조건

실패 조건은 다음과 같다.

| 실패 | 조건 |
| --- | --- |
| 조사 착수 | 감시 위험도 100 도달 |
| 자금 고갈 | 유동성 0 도달 |
| 가격 붕괴 | 가격이 하한선 이하로 하락 |
| 시장 정지 | 특정 문서 이벤트 후 강제 종료 |

MVP에서는 조사 착수, 자금 고갈, 가격 붕괴 3개를 우선 지원한다.

### 7.13 결과 정산

라운드 종료 후 Settlement Report를 표시한다.

표시 정보는 다음과 같다.

1. 성공/실패 여부
2. 최종 가격
3. 목표 대비 달성률
4. 남은 유동성
5. 최종 감시 위험도
6. 최고 여론
7. 사용한 카드 조합
8. 처리한 문서 이벤트 수
9. 풍자적 결과 문구
10. 최고 기록 갱신 여부

예시 결과 문구:

```text
수익은 남겼지만 시장은 망가졌다.
감시국은 아직 당신의 이름을 모른다.
```

### 7.14 로컬 기록

MVP에서는 브라우저 로컬 저장소에 기록을 저장한다.

저장 항목은 다음과 같다.

1. 최고 점수
2. 최고 생존 시간
3. 종목별 최고 결과
4. 마지막 설정
5. 튜토리얼 완료 여부

## 9. 비즈니스 규칙

### 9.1 허구 시장 원칙

1. 모든 종목은 허구다.
2. 실제 회사, 실제 코인, 실제 거래소, 실제 인물을 직접 참조하지 않는다.
3. 실제 금융 데이터를 사용하지 않는다.
4. 게임 내 가격은 내부 난수와 규칙으로만 생성한다.
5. 플레이어가 현실 투자 판단에 사용할 수 있는 정보를 제공하지 않는다.

### 9.2 안전한 표현 원칙

1. 실제 시세조종 절차를 단계별로 설명하지 않는다.
2. 실제 주문 방식, 계좌 운용, 거래소 우회, 불법 은폐 방법을 구체화하지 않는다.
3. 카드 효과는 가격 압력, 여론, 유동성, 감시 위험도 같은 추상 수치로 표현한다.
4. 결과 화면은 플레이어의 행위를 미화하기보다 풍자적으로 되돌려준다.
5. 게임 시작 또는 크레딧에 “본 게임은 허구의 풍자물이며 실제 투자·금융 행위와 무관하다”는 안내를 넣는다.

### 9.3 세션 진행

1. 세션은 ready 상태에서 시작한다.
2. 시작 후 running 상태가 된다.
3. 매 프레임 또는 고정 틱마다 가격, 자원, 압박 요소를 갱신한다.
4. 제한 시간이 끝나면 성공 조건을 평가한다.
5. 실패 조건이 먼저 충족되면 즉시 failed 상태가 된다.
6. 종료 후 Settlement Report를 생성한다.

### 9.4 가격 계산

1. 가격은 기본 변동성, 카드 효과, 압박 요소, 여론, 이벤트 효과에 의해 변한다.
2. 가격은 0 이하로 내려가지 않는다.
3. MVP에서는 현실적인 주문장 시뮬레이션을 구현하지 않는다.
4. 가격 변화는 플레이어가 이해할 수 있을 정도로 시각적으로 설명한다.
5. 급등락 시 뉴스 티커 또는 경고 UI를 표시한다.

### 9.5 카드 발동

1. 카드는 자동으로 발동한다.
2. 각 카드는 cooldownSeconds를 가진다.
3. 유동성이 부족하면 일부 카드는 발동하지 않거나 약화된다.
4. 카드 레벨이 오르면 효과 또는 주기가 개선된다.
5. 고위험 카드는 감시 위험도를 더 많이 올린다.

### 9.6 레벨업

1. 특정 경험치 또는 영향력 조건을 만족하면 레벨업 선택지가 표시된다.
2. 선택지는 기본 3개다.
3. 이미 보유한 카드는 강화 선택지로 등장할 수 있다.
4. 신규 카드는 보유 카드 수 제한 내에서 등장한다.
5. 선택 중에는 게임 시간을 일시정지할 수 있다.

### 9.7 감시 위험도

1. 감시 위험도는 0~100이다.
2. 100에 도달하면 라운드는 조사 착수 실패로 종료된다.
3. 감시 위험도가 높을수록 문서 이벤트 발생 확률이 증가한다.
4. 감시 위험도 70 이상에서는 경고 UI를 강화한다.
5. 감시 위험도를 낮추는 선택은 반드시 다른 비용을 가진다.

### 9.8 문서 이벤트

1. 문서 이벤트는 제한 시간 내 선택해야 한다.
2. 선택하지 않으면 기본 불리한 선택으로 처리할 수 있다.
3. 선택 결과는 즉시 효과와 지연 효과를 가질 수 있다.
4. 문서 이벤트는 플레이 흐름을 끊지 않도록 짧고 명확하게 작성한다.
5. 중요한 문서 이벤트는 결과 정산에 기록한다.

### 9.9 결과 기록

1. 세션 종료 시 SettlementRecord를 생성한다.
2. 최고 점수와 최고 생존 시간은 로컬 저장소에 저장한다.
3. 저장 실패 시 게임 플레이 자체는 중단하지 않는다.
4. 기록은 설정에서 초기화할 수 있어야 한다.

## 10. 운영 환경 및 제약사항

### 10.1 브라우저 기반 실행

본 제품은 1차 범위에서 데스크톱 브라우저 기반으로 만든다.

1. Chrome 계열 브라우저를 우선 지원한다.
2. Firefox와 Safari는 후속 호환성 검토 대상이다.
3. 모바일 브라우저 최적화는 1차 범위가 아니다.
4. 키보드와 마우스 조작을 기본으로 한다.
5. 화면 해상도는 1280x720 이상을 우선 고려한다.

### 10.2 JS/TS 기반 경량 개발

권장 기술 방향은 다음과 같다.

1. TypeScript
2. Phaser 3
3. Vite
4. JSON 기반 카드/이벤트 데이터
5. 브라우저 localStorage 또는 IndexedDB 기반 로컬 기록

MVP에서는 서버 없이 정적 파일로 실행 가능한 구조를 우선한다.

### 10.3 아트 및 사운드 제약

1. MVP는 저해상도 픽셀/도트풍 또는 제한된 팔레트 UI를 사용한다.
2. 캐릭터 애니메이션보다 UI 애니메이션과 문서 연출을 우선한다.
3. 사운드는 도장, 프린터, 경고음, 키보드, 차트 알림음 중심으로 구성한다.
4. 고품질 보이스, 컷신, 대형 일러스트는 후속 범위로 둔다.

### 10.4 법적·윤리적 표현 제약

1. 게임은 실제 금융 범죄를 조장하지 않아야 한다.
2. 실제 조작 방법을 튜토리얼처럼 설명하지 않아야 한다.
3. 풍자와 허구성을 명확히 해야 한다.
4. 스토어 출시 시 플랫폼 정책에 맞는 설명 문구를 검토해야 한다.

2. 효과
3. 부작용
4. 태그
5. 현재 빌드와의 시너지 표시

문서 이벤트 화면은 낡은 서류 또는 감시국 질의서처럼 보이게 만든다.

제공 요소는 다음과 같다.

1. 문서 제목
2. 발신 기관
3. 본문
4. 제한 시간
5. 선택지 버튼
6. 승인/반려/보류 도장 연출
7. 선택 결과 요약

### 11.4 결과 정산 화면

결과 정산 화면은 단순 점수판이 아니라 감시 보고서처럼 구성한다.

제공 요소는 다음과 같다.

1. 성공/실패 도장
2. 최종 가격 그래프
3. 수익/피해/감시 지표
4. 선택한 카드 목록
5. 처리한 문서 이벤트 목록
6. 풍자적 평가 문구
7. 다시 하기 버튼
8. 메인 메뉴 버튼

### 11.5 설정 화면

설정 화면은 최소한 다음을 제공한다.

1. 전체 음량
2. 효과음 음량
3. 화면 크기 또는 전체화면
4. 텍스트 속도
5. 흔들림 효과 켜기/끄기
6. 기록 초기화

1. 가격 틱 계산식과 각 변수의 기본 범위
2. 작전 카드 8개의 정확한 쿨다운, 효과량, 부작용
3. 압박 유닛 5종의 출현 주기와 스케일링
4. 문서 이벤트 8개의 선택지, 결과, 후폭풍 조건
5. 종목 3개의 초기값과 특수 규칙
6. 점수 계산식의 정확한 가중치
7. UI 팔레트, 폰트, 해상도 기준
8. localStorage 키 이름과 저장 실패 처리
9. 디버그 패널에서 노출할 상태값
10. 안전 문구의 최종 카피

## 15. 리스크 및 고려사항

### 15.1 실제 금융 범죄처럼 보일 위험

게임의 소재가 주가/코인 조작이므로, 실제 조작 방법을 안내하는 것처럼 보일 수 있다.

대응 방향은 다음과 같다.

1. 실제 종목, 거래소, 인물을 사용하지 않는다.
2. 조작 절차를 구체적 튜토리얼로 만들지 않는다.
3. 모든 효과는 추상 수치로 표현한다.
4. 결과 화면에서 풍자와 비판적 톤을 유지한다.
5. 게임 설명에 허구의 블랙코미디 풍자물임을 명시한다.

### 15.2 Vampire Survivors류 재미 부족

자동 발동 게임은 카드 조합이 약하면 방치형처럼 느껴질 수 있다.

대응 방향은 다음과 같다.

1. 레벨업 선택의 빈도를 적절히 유지한다.
2. 카드마다 명확한 강점과 부작용을 둔다.
3. 문서 이벤트로 중간중간 판단을 요구한다.
4. 감시 위험도와 유동성의 압박을 계속 유지한다.

### 15.3 Papers, Please 레퍼런스 과다 모방

문서·도장·관료제 UI가 너무 비슷하면 독창성이 약해질 수 있다.

대응 방향은 다음과 같다.

1. 입국 심사 세계관을 사용하지 않는다.
2. 국가, 여권, 검문소, 원작 캐릭터 구조를 쓰지 않는다.
3. 거래소 감시국, 작전 보고서, 차트 작전실이라는 별도 정체성을 만든다.
4. UI 레이아웃과 문구는 완전히 새로 설계한다.

### 15.4 차트 UI 가독성 문제

차트, 카드, 문서, 뉴스 티커가 동시에 나오면 화면이 복잡해질 수 있다.

대응 방향은 다음과 같다.

1. MVP에서는 정보 수를 제한한다.
2. 가장 중요한 가격, 유동성, 감시 위험도를 크게 표시한다.
3. 문서 이벤트 중에는 배경 정보를 약하게 처리한다.
4. 색상과 아이콘 규칙을 일관되게 유지한다.

### 15.5 밸런싱 난이도

가격, 여론, 유동성, 감시 위험도가 서로 연결되어 밸런싱이 어려울 수 있다.

대응 방향은 다음과 같다.

1. P0에서는 공식과 변수를 단순하게 유지한다.
2. 카드와 위협 수를 제한한다.
3. 디버그 패널로 상태값을 확인할 수 있게 한다.
4. 플레이테스트 후 수치 조정이 쉬운 JSON 데이터 구조를 사용한다.

### 15.6 소재의 호불호

시장 조작을 플레이어 역할로 삼는 설정은 일부 사용자에게 불쾌할 수 있다.

대응 방향은 다음과 같다.

1. 플레이어 행위를 영웅적으로 미화하지 않는다.
2. 결과 화면에서 사회적 비용과 후폭풍을 드러낸다.
3. 전체 톤은 범죄 미화가 아니라 탐욕 풍자로 잡는다.
4. 마케팅 문구도 “따라 해보는 게임”이 아니라 “블랙코미디 생존 게임”으로 작성한다.

## 16. 개발팀에게

이 제품은 실제 투자 시뮬레이터가 아니다.  
또한 실제 시장 조작 방법을 학습시키는 게임도 아니다.

1차 목표는 차트, 자동 카드, 문서 이벤트를 결합해 **작고 빠르게 재미를 검증할 수 있는 시장 풍자 생존 게임**을 만드는 것이다.

개발 시 다음 원칙을 우선한다.

1. 게임 내 모든 시장과 종목은 허구로 만든다.
2. 실제 금융 데이터와 API는 사용하지 않는다.
3. 가격 공식은 현실 재현보다 게임 재미와 가독성을 우선한다.
4. 카드 효과는 추상 수치로 표현한다.
5. 감시 위험도는 핵심 긴장 요소로 유지한다.
6. 유동성은 무한 자원이 아니라 선택 비용으로 작동해야 한다.
7. 문서 이벤트는 짧고 명확해야 한다.
8. UI는 차트보다 문서형 작전실의 정체성이 살아야 한다.
9. P0에서는 기능 수를 늘리기보다 한 판의 완주 재미를 우선한다.
10. 밸런싱 가능한 값은 코드에 박지 말고 데이터로 분리한다.
11. 로컬 저장 실패가 게임 실패로 이어지지 않게 한다.
12. 실제 범죄 방법을 설명하는 텍스트는 넣지 않는다.
13. 결과 화면은 수익만 보여주지 말고 후폭풍과 풍자를 함께 보여준다.
14. MVP는 작게 만들되, 카드·이벤트·종목 추가가 쉬운 구조로 만든다.
