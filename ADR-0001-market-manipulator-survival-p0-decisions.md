## 1. Context

PRD v0.1은 게임의 큰 방향을 정의했지만, P0 구현으로 바로 이어가기에는 다음 항목이 모호했다.

1. TypeScript와 순수 JavaScript 중 무엇을 P0 기준으로 삼을지 불명확했다.
2. Phaser 내부 렌더링과 DOM/Canvas 혼합 방식의 경계가 없었다.
3. localStorage와 IndexedDB 중 MVP 저장 방식을 결정하지 않았다.
4. 세션 길이가 10~20분 범위로 열려 있어 밸런싱 기준이 흔들릴 수 있었다.
5. 작전 카드, 압박 유닛, 문서 이벤트의 최소 수량이 확정되지 않았다.
6. 승리 조건과 실패 조건이 구체적이지 않았다.
7. 점수 계산이 수익 중심인지, 감시/윤리 비용을 포함하는지 불명확했다.
8. Papers, Please풍 디자인의 참고 범위와 과다 모방 방지 기준이 부족했다.
9. 주식/코인 조작 소재가 실제 금융 범죄 안내처럼 보이지 않도록 하는 표현 경계가 더 필요했다.

P0의 목적은 큰 시스템을 만드는 것이 아니라, 한 판의 재미와 톤을 검증하는 작고 완주 가능한 웹 MVP를 만드는 것이다.

---

## 2. Decision

다음 결정을 P0 기준으로 채택한다.

### 2.1 기술 스택

P0 구현 기준은 **TypeScript + Phaser 3 + Vite**로 한다.

순수 JavaScript는 버리고, 타입 정의가 있는 데이터 중심 구조를 사용한다.

### 2.2 렌더링 방식

P0 게임 화면은 **Phaser 단일 렌더링**을 우선한다.

차트, 작전 카드, 압박 유닛, 레벨업 선택, 문서 이벤트, 결과 정산은 Phaser Scene 안에서 처리한다.

DOM은 다음 경우에만 사용한다.

1. 개발용 디버그 패널
2. 접근성 보조 텍스트
3. 임시 빌드 정보 표시

### 2.3 저장 방식

P0 저장 방식은 **localStorage**로 한다.

저장 대상은 다음으로 제한한다.

1. 설정
2. 최고 기록
3. 최근 정산 기록
4. 음량 또는 화면 옵션

세션 중 상세 로그, 리플레이, 통계 히스토리는 P0에서 저장하지 않는다.

### 2.4 세션 범위

P0 세션은 **10분 단일 모드**로 한다.

15분/20분 모드, 난이도별 시간 조정, 일일 챌린지는 P0에서 제외한다.

### 2.5 콘텐츠 수량

P0 콘텐츠 수량은 다음으로 고정한다.

| 콘텐츠 | P0 수량 |
| --- | ---: |
| 허구 종목 | 3개 |
| 작전 카드 | 8개 |
| 압박 유닛 | 5종 |
| 문서 이벤트 | 8개 |
| 이벤트 선택지 | 이벤트당 2~3개 |

### 2.6 승리 조건

승리 조건은 다음 두 조건을 모두 만족하는 것이다.

1. 10분 동안 실패 조건에 도달하지 않는다.
2. 종료 시점에 목표 가격 밴드에 도달했거나 일정 시간 이상 유지했다.

### 2.7 실패 조건

실패 조건은 다음 중 하나다.

1. 감시 위험도 100 도달
2. 유동성 0 도달
3. 가격 붕괴선 이탈

### 2.8 점수 원칙

점수는 단순 수익률이 아니라 다음 요소를 합산한다.

1. 수익률
2. 잔여 감시 위험도 보너스
3. 잔여 유동성 보너스
4. 사회적 비용 패널티

이 결정은 플레이어가 수익만 극단적으로 높이는 전략을 최적해로 받아들이지 않게 하기 위한 것이다.

### 2.9 표현 안전 원칙

게임은 실제 금융 범죄를 안내하지 않는다.

P0 텍스트와 시스템은 다음 기준을 따른다.

1. 실제 종목, 실제 거래소, 실제 인물, 실제 사건을 사용하지 않는다.
2. 실제 주문 방식, 조작 절차, 회피 절차를 단계적으로 설명하지 않는다.
3. 작전 카드 효과는 추상 수치와 풍자 문구로 표현한다.
4. 결과 정산은 수익뿐 아니라 후폭풍과 사회적 비용을 함께 보여준다.
5. 메인 메뉴, 로딩 화면, 결과 정산 화면에 허구 풍자물 고지를 노출한다.

### 2.10 UI 정체성

Papers, Please는 참고 감성으로만 둔다.

P0 UI의 독자적 정체성은 다음으로 정의한다.

1. 거래소 감시국
2. 작전 보고서
3. 차트 작전실
4. 감시 경고문
5. 뉴스 티커
6. 낡은 CRT와 저채도 문서 패널
7. 도장/프린터/경고음 중심의 피드백

입국 심사, 여권, 국경, 원작 캐릭터, 원작 문구, 원작 레이아웃은 사용하지 않는다.

## 3. Consequences

### 3.1 Positive Consequences

1. 구현 기준이 작아지고 명확해진다.
2. Phaser Scene 중심으로 상태 관리가 단순해진다.
3. TypeScript로 카드, 이벤트, 종목 데이터를 안정적으로 다룰 수 있다.
4. localStorage만 사용하므로 MVP 저장 구현이 빠르다.
5. 10분 단일 모드로 밸런싱 범위가 좁아진다.
6. 콘텐츠 수량이 고정되어 P0 완료 기준이 명확해진다.
7. 안전 표현 원칙이 제품 문서와 구현 모두에 적용된다.
8. 레퍼런스 과다 모방 리스크를 줄인다.

### 3.2 Negative Consequences

1. DOM 기반 UI보다 Phaser UI 구현 비용이 조금 더 클 수 있다.
2. localStorage는 상세 통계와 대량 기록에 적합하지 않다.
3. 10분 단일 모드는 반복 플레이 다양성이 부족할 수 있다.
4. 실제 시장 데이터와 거리를 두므로 금융 시뮬레이션을 기대하는 사용자에게는 깊이가 낮아 보일 수 있다.
5. 안전 표현 원칙 때문에 일부 카드명과 이벤트 연출이 덜 자극적으로 느껴질 수 있다.

### 3.3 Mitigations

1. UI 컴포넌트는 Phaser 내부에서 단순한 패널/텍스트/버튼으로 제한한다.
2. 밸런싱 값은 코드에 박지 않고 JSON 또는 TypeScript 데이터 객체로 분리한다.
3. IndexedDB는 P2에서 상세 기록이 필요할 때 도입한다.
4. 10분 모드가 안정화된 뒤 15분/20분 모드와 난이도 선택을 검토한다.
5. 안전 표현은 재미를 죽이는 제한이 아니라 블랙코미디 정체성으로 흡수한다.

---

## 4. Alternatives Considered

### 4.1 순수 JavaScript

장점은 초기 진입이 빠르다는 점이다.  
하지만 카드, 이벤트, 종목, 점수 계산식이 늘어나면 데이터 오류를 잡기 어렵다.  
따라서 P0 기준에서는 채택하지 않는다.

### 4.2 React/DOM UI + Canvas 차트 혼합

문서 UI를 만들기는 쉽다.  
하지만 게임 루프, 입력, 애니메이션, 상태 동기화가 복잡해질 수 있다.  
P0에서는 Phaser 단일 렌더링을 우선한다.

### 4.3 IndexedDB 우선

상세 기록 저장에는 더 적합하다.  
하지만 P0의 저장 대상은 최고 기록과 설정 중심이므로 localStorage로 충분하다.

### 4.4 15~20분 세션

장기 빌드업은 더 풍부할 수 있다.  
하지만 첫 밸런싱 비용이 커진다.  
P0에서는 10분 단일 모드로 제한한다.

### 4.5 실제 시장 데이터 사용

현실감은 높아질 수 있다.  
하지만 법적·윤리적 리스크와 구현 복잡도가 커지고, 제품의 풍자적 추상화 방향과 맞지 않는다.  
P0뿐 아니라 기본 제품 방향에서도 제외한다.

## 5. Implementation Notes for SRS/SDD

SRS 또는 SDD에서는 이 ADR을 기준으로 다음 항목을 수치화한다.

1. `MarketDefinition` 3종의 초기 가격, 목표 밴드, 붕괴선, 변동성 계수
2. `OperationCard` 8개의 쿨다운, 효과량, 부작용, 레벨업 규칙
3. `PressureUnit` 5종의 출현 조건, 영향 대상, 스케일링
4. `InspectionEvent` 8개의 선택지, 즉시 효과, 지연 효과
5. 가격 틱 계산식
6. 점수 계산식
7. localStorage key와 schema version
8. Phaser Scene 구조
9. 안전 문구 최종 카피
10. 디버그 패널 노출값

## 6. Status

Accepted.

이 ADR은 PRD v0.1.1의 P0 기준 결정으로 적용한다.
후속 문서에서 이 결정을 변경하려면 새 ADR을 작성하거나 본 ADR을 Superseded 상태로 전환한다.
